# Adaptive Runtime Acceptance Summary

## Scope

This document summarizes Phase 3C.1 acceptance checks for CyberGuard Adaptive Learning Intelligence. It records safe verification findings only. It does not include API keys, full prompts, raw learner records, raw assessment answers, raw scenario decisions, or personally identifying learner data.

## Correct Start Commands

Backend:

```powershell
npm --prefix server start
```

Frontend:

```powershell
npm --prefix client start
```

Combined root development command:

```powershell
npm run dev
```

The server package currently has `start`, not `dev`. The root package provides `dev`, `dev:server`, and `dev:client`.

## Integration Path Audited

Adaptive context is generated by `server/src/adaptive/adaptiveLearning.service.js`, using deterministic rules from `adaptiveLearning.rules.js`.

Adaptive request detection is handled by `shouldUseAdaptiveLearning()` and `evaluateAgenticEligibility()`. Requests such as “What should I study next?” and “Which topic should I improve?” are eligible. Static explanation prompts such as “What is phishing?” are not eligible. Mutating, ownership-breaking, or prompt-injection requests are excluded.

The context reaches the controlled Agentic planner through `controlledAgentic.service.js` as a bounded model-facing summary. It reaches final CyberGuard generation through `ai.service.js`, which appends the controlled Agentic context to the route/context section for the final provider call.

The adaptive summary is capped to 1,800 characters. It does not consume the single controlled tool allowance. The maximum controlled Agentic tool execution remains one, and the planner path still uses OpenAI only.

## Representative Learner States

### State A: New Learner

Fixture shape:

- minimal or partial profile;
- no completed assessment;
- no progress;
- no scenario history;
- no active recommendation.

Observed expected behavior:

- signal quality is low;
- no strength is invented;
- topic support needs are `insufficient_data`;
- recommended next step can suggest completing the initial assessment;
- missing data is acknowledged.

### State B: Consistent Foundation Support Need

Fixture shape:

- beginner/developing assessment evidence for one topic;
- low mastery for the same topic;
- recent developing scenario evidence;
- active recommendation for the same topic.

Observed expected behavior:

- the topic becomes a support priority;
- support need is `reinforce_foundation` or `guided_practice`;
- confidence is medium/high depending on available signals;
- learner-safe wording is used.

### State C: Consistent Strength

Fixture shape:

- intermediate/advanced assessment evidence;
- strong mastery;
- recent proficient/strong scenario evidence.

Observed expected behavior:

- the topic can appear as a strength;
- CyberGuard may offer challenge-style guidance;
- no guaranteed improvement or permanent ability claim is made.

### State D: Conflicting Signals

Fixture shape:

- strong assessment evidence;
- latest scenario needs review;
- moderate or stale progress evidence.

Observed expected behavior:

- confidence is reduced;
- conflict reason is represented internally;
- support need prefers guided practice;
- the topic is not labeled as a simple strength.

### State E: Stale Data

Fixture shape:

- completed evidence older than 45 days;
- little recent activity.

Observed expected behavior:

- recent activity is marked `stale`;
- confidence is lowered;
- current capability is not asserted as certain;
- fresh practice can be suggested.

## Non-Mutation Audit

Automated Phase 3C tests use fixtures and fake repositories to verify that the adaptive service calls only `loadLearnerContextData()` and does not call write methods. The controlled Agentic integration test verifies that adaptive context does not add a second tool execution.

No database schema changes are part of Phase 3C.1. No RAG ingestion is run.

The opt-in live script is intentionally safe by default. Without `AI_LIVE_TEST=1`, it skips and makes no provider request. With the flag enabled, it uses bounded fixture context and records only safe metadata.

## Real Provider Verification Status

Implemented:

```powershell
npm --prefix server run test:adaptive-live
```

Default status: skipped safely unless `AI_LIVE_TEST=1` is explicitly set.

Live OpenAI execution was not run during this documentation update. When enabled, the script verifies:

- OpenAI is the planner provider;
- Gemini is not used;
- ILMU is not used for production planning;
- model request count is bounded;
- tool execution count is bounded;
- no secrets or raw learner records are printed.

## Browser Verification Status

Browser verification remains pending unless run manually or through browser automation.

Manual checks should cover:

- `#/admin/ai-agentic` at 1440px, 1024px, 768px, and 390px;
- English, Malay, and Simplified Chinese labels;
- Controlled Agentic Runtime section;
- Adaptive Learning Intelligence section;
- no learner records, keys, raw prompts, or horizontal overflow;
- learner CyberGuard prompts for adaptive, static, greeting, and low-data cases.

## Known Limitations

- Adaptive intelligence is deterministic support, not predictive modeling.
- No persisted adaptive recommendation is created.
- No automatic difficulty or score change exists.
- Browser verification is not claimed unless separately performed.
- Live provider verification requires explicit opt-in.
- Resource completion tracking is not available, so adaptive next steps cannot use completed Resource history.
